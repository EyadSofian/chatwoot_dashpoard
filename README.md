# تحليلات إنجوسوفت — Engosoft Chatwoot Analytics

لوحة تقارير عربية (RTL) لخدمة عملاء Chatwoot الخاصة بإنجوسوفت.

An Arabic-first, right-to-left analytics dashboard for a self-hosted Chatwoot
instance. It ingests Chatwoot conversations and messages, computes the metrics
Chatwoot itself does not expose — **response time measured from assignment**,
open/resolved duration across reopens, campaign reply correlation, and
Fahd/Botpress handoff outcomes — and serves them as filterable reports with CSV
export.

It is a **read-only analytics layer**. It never writes back to Chatwoot.

---

## What it reports

| Screen | Arabic | What it answers |
| --- | --- | --- |
| Overview | نظرة عامة | Volume, response time, SLA health, campaign and bot share at a glance |
| Agents | الموظفون | Conversations per agent per day, response time from assignment, status mix, resolved duration |
| Departments | الأقسام | Same, grouped by Sales / Operations / Complaints |
| Conversations | المحادثات | Searchable conversation list; a drawer shows the full timeline |
| Campaigns | الكامبينات | Jobs from both campaign apps, recipients, replies, and **`operatorName` — مين عمل الكامبين** |
| SLA | مستوى الخدمة | First-response and resolution breaches against configurable targets |
| Fahd Bot | بوت فهد | Bot handoffs: who was queued, who got a human reply, and how fast |
| Exports | التصدير | CSV export of every dataset with the active filters applied |
| Settings | الإعدادات | Webhook URL, backfill, connection tests, SLA and business hours |

Every screen shares one global filter bar: date range, department, team, agent,
status, inbox, campaign source, campaign, SLA state, and "needs reply only".

## How the metrics are calculated

This is the part that matters — the numbers are only trustworthy because of
what they **exclude**.

### Response time is measured from assignment, not from the customer's message

Chatwoot's own "first response time" starts at the customer's message. That
punishes agents for queue time they do not control. This app measures the
interval an agent is actually accountable for:

```
start = the moment the conversation was assigned to the agent
end   = that agent's first PUBLIC OUTGOING message after the assignment
```

- **Reassignment opens a brand-new interval.** If a conversation moves from
  agent A to agent B, agent A's interval is closed at the handoff and B gets a
  fresh clock starting at their own assignment. A reply from A that lands after
  the handoff is not credited to A's interval.
- A reply only counts if it comes **from the assigned agent** — a different
  agent chiming in does not close the interval.
- If an agent was assigned and never replied, the interval is recorded with
  `responded: false` and a null duration, so "never answered" stays visible
  instead of silently vanishing from the average.
- The conversation-level response time is the first interval that actually got a
  reply.

Assignment history comes from live `assignee_changed` webhooks stored in
`conversation_events`. Backfill cannot recover that history from the Chatwoot
API, so backfilled conversations fall back to a single interval starting at
conversation creation. **Response-time accuracy improves from the day the
webhook is connected onward.**

See `src/lib/metrics/assignment.ts` and `tests/assignment.test.ts`.

### What is NOT a human reply

A message closes a response interval **only** if it is public, outgoing, and
sent by an identifiable human agent. Everything below is excluded:

| Excluded | How it is detected |
| --- | --- |
| Private notes | `private: true` |
| **Fahd / Botpress** | `sender.type` contains `agent_bot`, or `sender.id` is listed in `BOT_AGENT_IDS` |
| Automation-rule messages | `content_attributes.automation_rule_id` is present |
| WhatsApp campaign templates | `message_type: 3`, or `content_attributes.template_params` is present |
| Native Chatwoot campaign sends | `content_attributes.campaign_id` is present |
| System messages | outgoing and public but with **no sender** — Chatwoot builds automation and campaign messages with a nil sender, so a missing sender is machine-generated by definition |
| Incoming / activity messages | `message_type` 0 / 2 |

This is why "Fahd answered instantly" never shows up as a 3-second agent
response. See `src/lib/metrics/humanReply.ts` and `tests/classifier.test.ts`.

### Open / resolved duration survives reopens

A conversation's lifetime is split into **segments**. A resolve closes the
current segment; a reopen starts a new one. A conversation resolved at 10:00,
reopened at 11:00, and resolved again at 11:30 has two segments (1h + 30m), not
one 1.5h span. The trailing segment of a still-open conversation has no
duration. See `src/lib/metrics/resolution.ts`.

### How campaigns are correlated

The campaign uploader (`chatwoot-campain-uploder`) stamps markers on the
conversation's `custom_attributes`. A conversation is treated as a campaign
conversation if it carries a campaign label **or** any `api_sent_*` marker:

```
api_campaign_label              api_campaign_reply_assign_mode
api_campaign_status             api_campaign_reply_team_id
api_campaign_created_at         api_campaign_reply_team_name
api_campaign_marked_at          api_campaign_reply_inbox_id
api_campaign_active_until       api_campaign_reply_pending
last_api_campaign_label         api_campaign_reply_assigned_at
last_api_template               api_campaign_reply_assignee_id
api_sent_<label>_<template>     api_campaign_reply_assignee_name
```

From these the app derives, per conversation: the campaign label and template,
the customer's reply, and the **first human agent reply after that** (never the
automation or the template send itself). `api_campaign_reply_team_id` maps the
conversation back to the Sales or Operations campaign app.

These names must match production byte for byte — if one drifts, campaign
reporting silently reports zero. `tests/campaign.test.ts` pins all of them.

### Duplicate webhooks cannot double-count

Every delivery is hashed (`sha256(event + rawBody)`) into `raw_events.dedupeKey`,
which is `UNIQUE`. A redelivery collides, is answered `{ duplicate: true }`, and
never reaches the metric engine. Derived lifecycle events carry their own
dedupe key. Beyond that, metrics are **recomputed in full** from the
conversation's messages rather than incremented, so even a forced reprocess
converges to the same numbers. See `tests/webhook.test.ts`.

## Running locally

Requires Node 20+ and a PostgreSQL database.

```bash
npm install
cp .env.example .env        # then fill in the values
npx prisma migrate deploy   # create the schema
npm run dev                 # http://localhost:3000
```

Sign in with `APP_USERNAME` / `APP_PASSWORD`. The dashboard is the first screen —
there is no marketing page.

The app starts empty. To fill it, either connect the webhook (live data from now
on) or run a backfill (history) — see below.

### Tests, typecheck, build

```bash
npm test          # vitest — metric engine, classifier, campaign correlation, idempotency
npm run typecheck # tsc --noEmit
npm run build     # production build
npx prisma validate
```

Copy `.env.example` to `.env`. Every variable, and what breaks without it:

| Variable | Required | Purpose |
| --- | --- | --- |

| `CHATWOOT_BASE_URL` | **yes** | e.g. `https://chat.engosoft.com`, no trailing slash |
| `CHATWOOT_ACCOUNT_ID` | **yes** | Numeric account id |
| `CHATWOOT_API_TOKEN` | **yes** | A Chatwoot **user access token** (Profile → Access Token) with admin/agent scope. Used for backfill and conversation fetches. |
| `WEBHOOK_SECRET` | strongly recommended | Authenticates the Chatwoot webhook. If unset, the endpoint accepts unauthenticated posts. |
| `APP_USERNAME`, `APP_PASSWORD` | **yes** | The single manager login for the dashboard |
| `SESSION_SECRET` | optional | Signs the session cookie. Derived from the other secrets if unset. |
| `CAMPAIGN_SALES_APP_URL` | optional | Base URL of the Sales campaign uploader app |
| `CAMPAIGN_OPERATIONS_APP_URL` | optional | Base URL of the Operations campaign uploader app |
| `CAMPAIGN_APPS_API_SECRET` | optional | Sent to both campaign apps as `x-webhook-secret` |
| `TIMEZONE` | optional | Defaults to `Africa/Cairo`. Drives day bucketing and business hours. |
| `SLA_FIRST_RESPONSE_MINUTES`, `SLA_RESOLUTION_HOURS` | optional | Initial SLA targets; editable in Settings afterwards |
| `BUSINESS_START`, `BUSINESS_END`, `BUSINESS_DAYS` | optional | Business hours window used for business-hours SLA |
| `BOT_LABEL`, `BOT_INBOX_IDS`, `BOT_AGENT_IDS` | optional | Fahd/Botpress detection. **`BOT_AGENT_IDS` is the important one** — any Chatwoot user id listed here is treated as a bot and its messages never count as human replies. |
| `DEPARTMENT_SALES_TEAM_ID`, `DEPARTMENT_OPERATIONS_TEAM_ID`, `DEPARTMENT_COMPLAINTS_TEAM_ID` | optional | Team → department mapping, used when the `engosoft_department` attribute is missing |
| `CRON_SECRET` | optional | Lets an external scheduler call `/api/backfill` and `/api/sync/campaigns` with `Authorization: Bearer <secret>` instead of a session |

Never commit real values. `.env` is gitignored; `.env.example` holds placeholders only.

## Deploying to Railway

1. **Create the project** from this GitHub repo. Railway detects `railway.json`
   and `nixpacks.toml` (Node 20 + OpenSSL, which Prisma's engines need).
2. **Add a PostgreSQL plugin.** Railway sets `DATABASE_URL` on the service
   automatically — do not paste it by hand.
3. **Set the remaining variables** from the table above in the service's
   Variables tab.
4. **Deploy.** The start command is
   `npx prisma migrate deploy && npm run start`, so the schema is created or
   migrated on every boot. Health checks hit `/api/health`, which reports which
   config is present without touching the database.

A `Dockerfile` is also included if you would rather deploy a container.

### After the first deploy

1. Open **Settings (الإعدادات)** and confirm every row of حالة الإعدادات is
   green, using the **اختبار** buttons to test the Chatwoot and campaign-app
   connections.
2. Connect the webhook (below).
3. Run a backfill to load history.

## Chatwoot webhook setup

In Chatwoot: **Settings → Integrations → Webhooks → Add new webhook.**

- **URL:** `https://<your-app>.up.railway.app/api/webhooks/chatwoot?token=<WEBHOOK_SECRET>`
- **Events:** subscribe to
  `conversation_created`, `conversation_updated`, `conversation_status_changed`,
  `conversation_resolved`, `conversation_reopened`, `assignee_changed`,
  `message_created`, `message_updated`.

The Settings screen shows the exact URL for your deployment with a copy button.

`assignee_changed` is the one you cannot skip — it is the only source of the
assignment timestamps that response time is measured from.

The endpoint accepts three auth schemes; any one passing is enough:

- `?token=` or `?secret=` query parameter
- `x-webhook-secret` header
- HMAC: `x-chatwoot-signature` = `sha256(<timestamp>.<rawBody>)` with
  `x-chatwoot-timestamp` (set `WEBHOOK_MAX_AGE_SECONDS` to reject stale ones)

The webhook always answers `200`, even on an internal error, so Chatwoot does
not enter a retry storm. Raw payloads are stored, so anything that failed to
process can be reprocessed later.

## Backfill

The webhook only captures events from the moment it is connected. To load
history, use **Settings → جلب البيانات** (7 / 30 / 60 / 90 days), or:

```bash
curl -X POST https://<your-app>/api/backfill \
  -H "Authorization: Bearer $CRON_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"days": 30}'
```

Body: `{ days: 1–365, maxPages?, maxConversations? }`. It pages through the
Chatwoot API, fetches each conversation's full message list, recomputes every
metric, and upserts. It is **safe to re-run** — nothing double-counts.

Backfill reconstructs message-derived metrics exactly, but cannot recover
assignment or reopen history that predates the webhook (Chatwoot's API does not
expose it). Those conversations get a single assignment interval from creation.

## Self-healing drift repair

The dashboard is a **mirror** of Chatwoot, kept fresh by webhooks. Any missed
delivery (webhook added late, app redeploy, Chatwoot retry exhausted) leaves
rows stale: a conversation reassigned in Chatwoot still shows the old agent,
and that agent's current workload counts it. Two mechanisms fix this:

1. **Heal on touch** — every webhook event (including `message_created`)
   re-fetches the full conversation from Chatwoot and upserts it, so any drift
   on an *active* conversation disappears with its next event.
2. **Reconcile sweep** — `POST /api/audit/reconcile-current-workload` pages
   Chatwoot's open/pending/snoozed lists, diffs them against the mirror
   (missing rows, different assignee, different status, phantom-active rows),
   and re-ingests every mismatch. The **Accuracy Audit** page has a one-click
   button that loops until the diff is empty.

Schedule both so drift never accumulates (Railway Cron or any scheduler):

```bash
# nightly at 03:00 — refresh the last 2 days of history
curl -X POST https://<your-app>/api/backfill \
  -H "Authorization: Bearer $CRON_SECRET" \
  -H "Content-Type: application/json" \
  -d '{"days": 2, "scope": "recent"}'

# nightly at 03:30 — repair any assignee/status drift on active conversations
curl -X POST https://<your-app>/api/audit/reconcile-current-workload \
  -H "Authorization: Bearer $CRON_SECRET"
```

If **Last webhook** in the sidebar (or the red banner on the Webhook card in
Settings) says no event has ever arrived, the webhook is not connected — fix
that first; no amount of reconciling keeps up with a dead webhook.

## Campaign apps integration

The two uploader apps (`chatwoot-campain-uploder`, one Sales, one Operations)
are polled over HTTP. Each must expose:

- `GET /api/jobs?limit=N` — job summaries
- `GET /api/jobs/:id` — one job's detail, including sent/failed recipients

Requests carry `x-webhook-secret: <CAMPAIGN_APPS_API_SECRET>` when that variable
is set. Point `CAMPAIGN_SALES_APP_URL` and `CAMPAIGN_OPERATIONS_APP_URL` at them
and sync with **Settings → مزامنة الكامبينات**, or:

```bash
curl -X POST https://<your-app>/api/sync/campaigns \
  -H "Authorization: Bearer $CRON_SECRET"
```

Sync stores each job with its `operatorName` — **مين عمل الكامبين**, the person
who actually launched the campaign — which is shown on the Campaigns table, in
the job drawer, and in the CSV export. Campaign *replies* are correlated
separately, from the Chatwoot custom attributes described above, so reply
metrics work even if an app is temporarily unreachable.

Both endpoints also accept a logged-in session, so a scheduler is optional —
set `CRON_SECRET` and point Railway Cron (or any scheduler) at them to keep the
data fresh automatically.

## SLA settings

**Settings → إعدادات SLA** configures:

- **First-response target** (minutes) — measured from assignment, per the rules above
- **Resolution target** (hours)
- **Business hours** — start, end, and working days

SLA is evaluated against **business hours** when a window is configured, so a
ticket that arrives at 23:00 is not marked breached at 09:00 the next morning.
A conversation is `near_breach` at 80% of target and `breached` past it. Open
conversations are evaluated against elapsed time, so a breach surfaces while it
is still actionable rather than only after the fact.

Settings are stored in the database (`app_settings`) and take effect on the next
recompute; the env vars only seed the initial values.

## Project layout

```
src/
  app/
    (dashboard)/        Arabic RTL screens — / is the Overview, not a landing page
    api/                REST routes (thin handlers over lib/)
    login/
  components/           Shell, filter bar, tables, charts, drawers
  lib/
    chatwoot/           API client, fetchers, webhook verification
    campaigns/          Campaign-app client
    metrics/            The engine: humanReply · assignment · resolution · campaign · conversation
    ingest/             webhook · backfill · campaignSync · persist (all idempotent)
    reporting/          Query layer behind each report route
prisma/schema.prisma    PostgreSQL schema
tests/                  Vitest suites for the engine
```

The metric engine in `src/lib/metrics/` is pure — it takes a Chatwoot
conversation plus its messages and returns computed records, with no database
access. That is what makes it directly testable, and it is where the exclusion
rules above live.
